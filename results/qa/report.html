핵심 증명 — CMS 편집이 랜딩에 반영

관리자 패널에서 히어로 제목을 "관리자가 방금 수정한 제목입니다 — E2E 반영 테스트"로 바꿔 저장하자, 공개 랜딩페이지에 그대로 노출됨. 즉 비개발자가 코드/배포 없이 콘텐츠를 직접 바꿀 수 있다는 CMS의 핵심 가치가 라이브에서 실증됨. (검증 후 원본 제목으로 원복 완료 — 라이브 클린)

전체 스크린샷

실측 발견 / 주의사항

① D1 전파 지연 (eventual consistency)
콘텐츠 저장 직후 공개 /가 잠깐 이전 값을 보였다가 수초 내 새 값으로 수렴. 응답 헤더는 cache-control: private, no-cache, no-store라 HTTP/CDN 캐시는 아님 → 원격 D1 바인딩의 읽기 일관성 지연으로 추정. 운영상 "수정 즉시 100% 반영"은 아니고 수초 지연 가능. (편집 UX 평가에 반영)
② 콜드 스타트
유휴 후 첫 요청 TTFB ~2s, 이후 웜 ~0.2s. 트래픽 적은 랜딩은 첫 방문자가 콜드스타트를 맞을 수 있음.
③ 이메일 어댑터 미설정
비밀번호 재설정 / 이메일 인증 / 폼 알림 메일은 어댑터(Resend·nodemailer) 미설정이라 실제 발송 안 됨(콘솔 출력만). 로그인 자체는 영향 없음. 폼/리드 알림까지 테스트하려면 어댑터 추가 필요.

정직성 — 자동화 우회 감사

모든 E2E 액션은 실제 사람이 브라우저에서 할 수 있는 행동(키보드 입력, 클릭, URL 입력)만 사용. 아래 false-green 유발 우회 전부 0건:
  • httpCredentials / extraHTTPHeaders Authorization 주입 — 0
  • storageState 사전 주입 / 쿠키·세션스토리지 직접 주입 — 0
  • URL-embedded credentials / API 직접 호출로 인증 상태 위조 — 0
관리자 첫 계정 생성과 로그인 모두 폼에 직접 타이핑 + 클릭으로 수행했으므로, "통과"는 실제 사용자도 동일 흐름을 완료할 수 있음을 의미함.

카테고리별 결론

  • 성능만: baseline = Keystatic = Sveltia (정적, Perf 100). CMS 종류 무관, 결과물이 정적이면 동일.
  • 비개발자 즉시 편집 + 폼/DB + CF 단일스택: Payload on Workers (Perf 97 감수, admin 바로 반영).
  • 정밀 권한·관계형·헤드리스: Directus(단 BSL+별도서버) vs PocketBase(MIT·초경량·별도서버). 라이선스/셋업은 PocketBase 우위.
